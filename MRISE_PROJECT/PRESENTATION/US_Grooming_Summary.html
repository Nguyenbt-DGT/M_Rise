M-Rise → AMS Sync Backlog grooming summary · 13 APIs · 4 Orders · prepared by BA
Grooming packet · US1–US4

Replacing two legacy e-learning systems with a 4-order sync into AMS

M-Rise is taking over class, exam, result, license, and e-certificate processing that two retiring systems used to handle, and pushing that data into the CAS system (AMS) through a middleware layer called the Learning Adapter. The 13 APIs in scope split into four dependent Orders — each one has to land before the next can be trusted to have the data it needs. This packet is the one-pager for grooming: objective, flow, acceptance criteria, and the open questions the team should actually talk about.

01 Class setup 3 APIs · template, exam, class
→
02 Candidate results 3 APIs · attendance, training, CPA
→
03 License & plan code 5 APIs · eligibility, grant, plan codes
→
04 E-certificate 2 APIs · save + retrieve

Each Order gates the next: Order 1's class/exam shell must exist before Order 2's results can attach to it; Order 2's results feed Order 3's license/plan-code grants; Order 3's granted license gates Order 4's e-certificate. All 13 APIs run M-Rise → Learning Adapter → AMS — M-Rise never calls AMS directly.

Reading this packet

Each Order tab has four parts: the objective (why it exists), the flow (trigger → pre-condition → post-condition), the acceptance criteria (what "done" means), and discussion flags — the open items, PO decisions, and assumptions the squad (PO/BA/BE) already surfaced while drafting. The amber flags are exactly what grooming is for: confirm them, challenge them, or accept them as logged risk.

01

Sync Class Setup Information — template, exam, class

Approved 3 APIs Blocks Orders 2, 3, 4
Objective As AMS System, I want to receive class setup information from M-Rise through Learning Adapter, so AMS can initialize the class template, exam information, and class information that candidate/agent learning management and every downstream Order depends on.
Trigger
  • Class template created/activated on M-Rise
  • Exam code successfully imported in a class
  • Training plan approved on M-Rise
Pre-condition
  • M-Rise has template/exam/class data ready
  • Learning Adapter is running
  • Related master data exists in AMS where required
Post-condition
  • AMS stores template, exam, and class records
  • Data is ready for Order 2 (attendance/result)
  • Every call is logged

Acceptance criteria

01

Sync class template successfully

Given a template is created/activated on M-Rise When Learning Adapter calls sync-class-template Then AMS inserts/updates/deletes by action, writes each plan code as its own row, and never duplicates on re-sync.

02

Sync exam information successfully

Given an exam code is imported on M-Rise When exam-info/sync is called Then AMS validates examTypeCd against TAMS_EXAMS, creates or updates the exam record, and rejects unknown exam types.

03

Sync class information successfully

Given a training plan is approved on M-Rise When sync-class is called Then AMS validates templateCode / classType exist, then inserts/updates/deletes the class, with no duplicates on re-sync.

04

Integration must go through Learning Adapter

AMS never receives an Order 1 request directly from M-Rise — every call is relayed by Learning Adapter.

05

Error handling & logging

Every call logs API name, timestamp, status, error message, and reference key (templateCode / examCode / classCode).

06

Validation failure handling

Missing required fields, invalid examTypeCd, missing templateCode/classType, or a blocked exam delete all reject cleanly with no partial writes.

Discussion flags for grooming

  1. How do we tell an M-Rise-origin record apart from other sources once it's inside AMS? Raised by the story's original author (Nguyen Bien Trung) and still open — needs a check with the AMS data-model owner before build. This is the one open item on an otherwise fully-approved story.Open
  2. mitType = 'F' special case: inserting a class with mitType = F makes AMS auto-create a linked sub-class (suffix _C, mitType = C) — confirm the team building sync-class is aware this isn't a 1-row-in-1-row-out API.Confirm build understanding
02

Sync Attendance, Exam Result & Training Result

Approved 3 APIs Depends on Order 1 · Blocks Orders 3, 4
Objective As AMS System, I want to receive candidate exam attendance/result and training-result information from M-Rise, so AMS can record exam attendance & result, class training result, candidate result, education/license background, MOF approval, and CPA-result-sent status for Order 3/4 processing.
Trigger
  • Candidate exam attendance/result recorded
  • Candidate training/class result finalized
  • Candidate result sent to the AP (CPA) system
Pre-condition
  • Order 1's class/exam/template shell exists
  • inform-cpa-result requires training-result/sync to have already run for the same candidate
Post-condition
  • Attendance, result, education background, MOF decision stored
  • "Sent to AP" status reflected
  • Data ready for Order 3

Acceptance criteria

01

Sync exam attendance & result

Insert-or-update by (EXAM_MASTER_CD, CLASS_CD, CAN_NUM) — no action field, existence check decides insert vs. update; no duplicates on re-sync.

02

Sync training result

Insert/update TAMS_TRAINING_RSLTS always; if isMIT, also write candidate result + education background; if not MIT but hasLDPLicense, education background only; MOF fields update independently whenever present.

03

Inform CPA result

Locate the candidate-result row and set its "sent to AP" flag; re-sending the same (classCode, candidateNum) is idempotent and does not error.

04

Integration must go through Learning Adapter

Same enforcement as Order 1 — no direct M-Rise → AMS calls for any of the 3 APIs.

05

Error handling & logging

Every call logs API name, timestamp, status, error, and reference key (classCode, canNum/candidateNum, examCode).

06

Validation failure handling

Missing required fields, missing Order 1 parent data, an incomplete MIT payload, or a duplicate education-background insert all reject or no-op cleanly — never a partial write.

Discussion flags for grooming

  1. Order placement of inform-cpa-result/sync: BA initially flagged this as possibly Order-3-adjacent. PO ruled it stays in Order 2 — it only flips a status flag on a row that Order 2 itself creates, with no Order 3 dependency. Worth a quick team sanity-check, not a re-litigation.PO decision — confirm understanding
  2. The "sent to AP" column name on TAMS_CANDIDATE_RSLTS is not in the source workbook — assumed to be a SEND_TO_AP-style flag. Needs a DDL/data-dictionary lookup before dev estimates R.03.Open — blocks estimation
  3. isMIT = true but a required sub-object is missing from the payload: PO approved fail-fast (reject the whole request) over a partial write. Revisit only if M-Rise's client team confirms a real phased-submission use case.PO decision
  4. When both isMIT and hasLDPLicense are false — are the sub-objects omitted entirely, or sent null/empty? Affects request-schema validation; needs confirmation from the M-Rise API client owner.Open
  5. Order 1 existence checks (classCode/examCode must already exist) are engineering inferences, not sourced validation rows — approved as default build behavior, but flag for SIT verification against real AMS behavior.Confirm in SIT
03

License Eligibility, Grant & Plan Code

Approved 5 APIs Depends on Order 2 · Blocks Order 4
Objective As AMS System, I want to receive license-eligibility, license-grant, and plan-code assignment/refresh information from M-Rise, so AMS can determine whether a candidate qualifies for a license, record the granted license/education background, and maintain the agent's product plan codes for Order 4.
Trigger
  • License-grant eligibility needs checking
  • A result qualifies a candidate and M-Rise finalizes the grant
  • A plan code needs assigning (new) or refreshing (expiry/status)
  • MIT candidate license/education needs updating
Pre-condition
  • Order 2's education background / candidate data exists
  • plancode/refresh requires the target row to already exist — it only updates, never inserts
Post-condition
  • Eligibility flags, granted license, AWS cert housekeeping reflected
  • Plan codes assigned/refreshed with no duplicates
  • Data ready for Order 4

Acceptance criteria

01

Check license-grant conditions

Read-only despite the POST verb — evaluates eligibility per education type (UL/ULC/PENSION) and returns a true/false map; never writes to AMS.

02

Refresh plan code

Update-only on (AGT_CD, PLAN_CD) — if the row doesn't already exist, AMS must not insert one.

03

Assign plan code

Insert a new TAMS_AGT_PRODUCT_DTLS row only if (AGT_CD, PLAN_CD) doesn't already exist — existence check is a hard build requirement (BR-012), not optional.

04

Grant license

Insert/update education background by (CAN_NUM, EDU_TYP); archive prior MIT/ULC cert under an _MVL suffix when applicable; delete the old AWS e-cert record; conditionally update product code and MOF approval. Reports success/failure via errorCode in the response body, not an HTTP error.

05

Update MIT license

Insert a new education-background row only if (CAN_NUM, EDU_TYP) doesn't already exist (BR-012); maps certDecNum/certDecDate to the MOF approval fields.

06

Integration must go through Learning Adapter

Same enforcement as Orders 1–2, across all 5 APIs.

07

Error handling & logging

Every call is logged, including license/grant's soft-failure errorCode — it must still be treated as a failure for logging/alerting even though no exception is raised.

08

Validation failure handling

Missing required fields, a not-found plan-code row on refresh, or a duplicate re-sync on the two insert-only APIs all reject or no-op cleanly.

Discussion flags for grooming

  1. Two APIs (plancode/assign, MIT-license/update) were sourced as bare inserts with no existence check — a real gap versus every other write API in this integration. PO made adding a defensive existence check a hard build requirement (BR-012), not optional, because this is candidate-facing license/plan-code data with real compliance consequence if duplicated on a Learning Adapter retry. Team should confirm the idempotent-response contract (return success on redundant retry vs. hard reject) during dev.PO decision — dev must implement
  2. MIT-license/update's field mapping looked shifted by one row in the source workbook (certDecNum/certDecDate/certCateId → MOF fields didn't line up with their own descriptions). BA and BE independently converged on a corrected mapping, and PO approved building against the correction rather than the literal sheet, since it writes MOF (financial regulator) approval fields. Still needs workbook-owner confirmation.Open — build proceeds on corrected mapping
  3. plancode/refresh sets STAT_CD in its SQL but the request schema documents no statusCode input field. Working assumption: AMS derives the status server-side. Confirm with the M-Rise API owner in SIT.Open — confirm in SIT
  4. license/grant's soft-failure pattern (errorCode: SUCCESS/FAILED in a 200-style response, not an HTTP reject) is genuinely different from every other API in this integration — make sure whoever builds logging/alerting for this story knows not to rely on HTTP status alone.Confirm build understanding
04

Save & Retrieve E-Certificate

Approved 2 APIs · 1 write + 1 read Depends on Orders 2 & 3 · Final Order
Objective As AMS System, I want to receive and record an agent's e-certificate document reference from M-Rise — and as Learning Adapter, I want to retrieve an agent's e-certificate eligibility and delivery configuration from AMS — so M-Rise can generate or (re-)deliver the correct certificate. The only Order in this integration with both a write and a read direction.
Trigger
  • E-certificate document generated/uploaded on M-Rise (save)
  • Eligibility/delivery-config lookup needed (get)
Pre-condition
  • Order 2 PASS training result exists
  • Order 3 active license/education record exists
  • No existing active matching cert (for save)
Post-condition
  • New housekeeping row saved, or a 406/404 rejection
  • Get returns eligibility + (partial) delivery data
  • Every call logged

Acceptance criteria

01

Save e-certificate

Reject 406 if an active matching cert already exists; reject 404 if the agent isn't eligible (no PASS result / no active license); otherwise insert, sourcing agent/education fields via join.

02

Retrieve e-certificate information

Read-only lookup by licNo + agtCode; returns education-background data now, delivery-configuration data once its schema is confirmed (see flags).

03

Integration must go through Learning Adapter

Applies in both directions — save and get.

04

Error handling & logging

Logs include the returnCode for save's 406/404 business-rejection cases, not just hard errors.

05

Validation failure handling

Missing required fields reject on both APIs; an ineligible agent on get returns an empty result rather than erroring — deliberately different from save's explicit reject.

Discussion flags for grooming

  1. The get endpoint's delivery-configuration response (cas.tdelivery_settings_dtls / tdoc_settings) has no field list anywhere in source. PO ruled the team should not invent one. This is a scoped pre-build blocker on that sub-piece only — the education-background half of the same API is fully specified and can proceed now. Needs the DDL from the workbook owner before that part can be built.Open — blocks one response sub-piece
  2. Two distinct rejection codes were needed for save (duplicate cert vs. agent-not-eligible) so Learning Adapter's retry logic doesn't conflate them. Source only documented 406; PO assigned 404 for the not-eligible case. Confirm against the actual M-Rise/AMS API-owner convention in SIT — it may need to change.PO decision — confirm in SIT
  3. Why does get silently return empty for an ineligible agent while save explicitly rejects? This is deliberate, not an inconsistency — a query with nothing to return shouldn't error, but a write with nothing to source shouldn't insert null housekeeping fields. Worth stating out loud in grooming so QA writes test cases for both behaviors correctly.Confirm build understanding
  4. LIC_STAT_CD is filtered on in the save existence-check but never introduced as a real field anywhere in the doc — same pattern as Order 3's STAT_CD gap. Working assumption: a fixed/derived value, not client-supplied. Confirm against the stored procedure/DDL during dev.Open — confirm in dev
